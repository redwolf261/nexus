# Provenance Model
**Phase 7.1 Analytical Intelligence**

## The Provenance Chain
Every analytical result generated by the NEXUS Intelligence Engine is traceable through a strict 6-stage provenance chain. Opaque "black-box" scores are forbidden.

```
[1] Observation
      ↓
[2] Evidence
      ↓
[3] Analytical Rule
      ↓
[4] Inference
      ↓
[5] Confidence Score
      ↓
[6] Recommended Action
```

## Schema Enforcement
This chain is enforced via the `IntelligenceExplanation` Pydantic model (`backend/intelligence/explainability.py`). Every endpoint in `/api/intelligence/*` returns this schema for every match, cluster, or anomaly.

### Example: Entity Resolution
1. **Observation:** "Entity C-123 scored against C-456"
2. **Evidence:**
   - Dimension: `phone_exact` (Raw: 9876543210, Weight: 0.20)
   - Dimension: `name_jaro_winkler` (Raw: 0.92, Weight: 0.15)
3. **Analytical Rule:** "Weighted multi-dimensional Jaro-Winkler"
4. **Inference:** "C-456 is a probable match for C-123 with score 0.35"
5. **Confidence:** 82.4% (Geometric Mean)
6. **Action:** "Cross-reference Aadhaar database"

## Frontend Integration
The frontend `IntelligencePanel` (introduced in Phase 7.1) iterates through the `evidence` array of the explanation object to render a transparent breakdown. Analysts can click on any intelligence card to view the exact math and rules that triggered the alert.

## Event Bus & WebSockets
When intelligence is discovered in the background (e.g. by `intelligence_worker.py`), the full `IntelligenceExplanation` is serialized and broadcast via WebSockets. The frontend Redux/React-Query cache hydrates using this exact provenance model, ensuring no data loss between backend discovery and UI rendering.
